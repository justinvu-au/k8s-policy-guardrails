# Kubernetes Policy Guardrails

Cluster-wide security and operational policies enforced via [OPA Gatekeeper](https://open-policy-agent.github.io/gatekeeper/), with automated offline testing and a CI/CD pipeline that validates and deploys policy changes to a live AKS cluster behind a manual approval gate.

---

## What it does

Enforces three rules on every pod scheduled to the cluster, independent of which application or team deployed it:

1. **No privileged containers** — blocks containers requesting elevated host access
2. **Resource limits required** — every container must declare CPU and memory limits
3. **`app` label required** — every pod must carry an `app` label for operational traceability

These are enforced at admission time (before a violating pod is ever created) and continuously re-checked by Gatekeeper's audit controller against everything already running.

---

## Architecture

```
kubectl apply (any pod, from any source)
        ↓
Kubernetes API Server
        ↓
Gatekeeper validating admission webhook
        ↓
Evaluates against installed Constraints (Rego policies)
        ↓
   Violates a policy? ──▶ REJECTED — pod never created
        │
      Compliant
        ↓
   Pod created normally
```

git push → main

↓

CI: gator test (offline, no cluster needed)

├─ good/ examples must pass with zero violations

└─ bad/ examples must each be correctly flagged

↓

Tests pass?

↓

CD: pauses for manual approval (GitHub Environment: production)

↓

Approved

↓

kubectl apply → policies live on plstats-aks

---

## Why a manual approval gate on the CD step

Applying cluster-wide policy is higher blast radius than deploying a single application — a misconfigured Rego rule could block legitimate workloads across every team's pods, not just one service's. Tests run automatically on every push and PR; only the actual `kubectl apply` to the live cluster requires a human to explicitly approve, mirroring the same pattern used for production deployments in [multi-env-infrastructure](#).

```
k8s-policy-guardrails/

├── policies/

│   ├── 01-block-privileged-containers.yaml

│   ├── 02-require-resource-limits.yaml

│   └── 03-require-app-label.yaml

├── examples/

│   ├── bad/

│   │   ├── privileged-pod.yaml          # violates policy 1 only

│   │   ├── no-resource-limits.yaml      # violates policy 2 only

│   │   └── missing-app-label.yaml       # violates policy 3 only

│   └── good/

│       └── compliant-pod.yaml           # passes all three — proves policies aren't over-broad

└── .github/

└── workflows/

└── validate-policies.yml

```
Each "bad" example is deliberately scoped to violate exactly one policy — this makes it possible to assert precisely which rule fired, rather than a vague "something failed."

## Running tests locally

```bash
brew install gator

# Good example must pass with zero violations
gator test -f policies/ -f examples/good/

# Bad examples must each be flagged
gator test -f policies/ -f examples/bad/
```

`gator` evaluates manifests against the Rego policies entirely offline — no cluster connection needed, making it fast and safe to run on every commit.

## Deploying to a live cluster

```bash
az aks get-credentials --resource-group pl-stats-rg --name plstats-aks
kubectl config current-context   # always verify before applying cluster-wide policy

kubectl apply -f policies/
# Constraints reference CRDs generated by their ConstraintTemplate — if applying
# for the first time, re-run this command once after a few seconds; the CRD
# needs a moment to register before the matching Constraint can attach to it.

kubectl get constrainttemplates
kubectl get constraints
```

## Known limitation: `failurePolicy: Ignore`

Gatekeeper's default validating webhook configuration uses `failurePolicy: Ignore` with a 3-second timeout. This means if the webhook is briefly slow, busy, or unavailable, **the API server allows the request through rather than blocking it** — enforcement is not unconditionally guaranteed.

This was discovered during testing: a deliberately non-compliant pod was created successfully despite an active, correctly-configured policy, shortly after a constraint update while Gatekeeper was still reconciling.

This is a deliberate trade-off, not a bug, and it's the configuration most real Gatekeeper deployments ship with: **prioritising cluster availability over airtight enforcement**. The alternative (`failurePolicy: Fail`) guarantees enforcement but means *no pods can be created at all* if Gatekeeper itself is ever down or overloaded — a worse outcome for most teams than an occasional missed violation, especially since the audit controller still detects (though doesn't block) anything that slips through. This repo keeps the default `Ignore` behaviour as a conscious choice, with the audit controller treated as a compensating control rather than relying on the webhook alone.

## Lessons learned

**Rego's handling of missing vs. empty fields.** The original `require-app-label` policy used `not provided[required]` where `provided := input.review.object.metadata.labels`. When a pod has no `labels` field at all (not even an empty map), `provided` is `undefined` in Rego, and the rule silently fails to match — meaning the violation was never raised for pods with zero labels, the most common case it should have caught. Fixed using `object.get(input.review.object.metadata, "labels", {})`, which safely defaults to an empty object instead of `undefined`. Found via the offline `gator` test suite, not by chance — direct evidence for why automated policy testing matters even for "simple" Rego rules.

**Test exit-code semantics need to be explicit.** `gator test` exits non-zero whenever it finds any violation, with no built-in concept of "this violation was expected." Running it against a folder of deliberately non-compliant examples will therefore always exit `1` — which looks like a CI failure even when the policies are working perfectly. The fix was splitting the CI step into two explicit assertions: one expecting zero violations (the good example), and one explicitly inverting the check — treating a *clean* `gator test` run against the bad examples as the actual failure condition, since it would mean a violation went undetected.

**Environment parity between local and CI.** The CI pipeline initially failed with `Exec format error` when installing `gator` — caused by downloading the `darwin-amd64` (macOS) binary in a workflow running on `ubuntu-latest` (Linux). Worked perfectly when tested locally on a Mac; failed immediately in CI. A direct, simple example of why "works on my machine" isn't sufficient validation for a CI/CD pipeline.

**`kubectl apply` does not handle resource *kind* changes.** While investigating an unrelated duplicate-pod issue on a different project sharing this cluster, discovered that converting a `Deployment` to an Argo Rollouts `Rollout` (same name, different `kind`) via `kubectl apply` left the original `Deployment` running untouched alongside the new `Rollout` — both independently managing pods with the same labels, silently doubling running pod count and cost for weeks. `kubectl apply` matches by name *and kind*; changing kind requires an explicit `kubectl delete` of the old resource first.

**Audit-discovered violations need deliberate scoping, not blanket exclusion or blanket enforcement.** Enabling the `app` label policy immediately flagged five pre-existing violations — all belonging to the Prometheus stack and Argo Rollouts, neither of which this project authors the manifests for. Rather than disabling the policy or patching third-party Helm charts, the constraint's `excludedNamespaces` was deliberately scoped to exclude `monitoring` and `argo-rollouts`, keeping enforcement focused on workloads actually owned by this project while leaving infrastructure components alone.
